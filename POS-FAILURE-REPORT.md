# Баг-репорт: «Заказ не доходит до кассы (POS)» — проблема №1 поддержки

_Источник: анализ всех ~92К сообщений поддержки Delever (2026-03 — 2026-06). Сгенерировано из данных._

## Суть

**Самая частая проблема поддержки** — заказы из агрегаторов **не попадают в кассовую систему ресторана**. По таксономии — **587 обращений (30% всех заказных тем)**. Углублённый разбор по 70 затронутым каналам выявил **884 отдельных упоминания инцидента**.

Клиенты пишут: _«Заказ не поступил в айко»_, _«заказы не поступают на кассу»_, _«iiko tushmagan»_, _«не поступают заказы на доставку»_.

## Где ломается

**POS-система (куда не доходит):**
| POS | Инцидентов | Упоминаний всего |
|---|---|---|
| **iiko** | 430 | 3 293 |
| 1c | — | 219 |
| poster | 4 | 136 |
| jowi | — | 117 |
| rkeeper | 10 | 26 |

→ Проблема почти целиком в связке с **iiko** (доминирующая POS у партнёров).

**Агрегатор (откуда заказ):**
| Маршрут | Инцидентов |
|---|---|
| yandex → iiko | 111 |
| delever(own) → iiko | 80 |
| uzum → iiko | 41 |
| wolt → iiko | 21 |
| (источник не определён) | ~340 |

→ Топ-источник — **Yandex**, затем собственные заказы Delever, Uzum, Wolt.

## Чья вина (по собственной диагностике поддержки)

| Сторона | Инцидентов | % |
|---|---|---|
| **Delever (интеграционный слой)** | **531** | **60%** |
| Агрегатор | 181 | 20% |
| POS-система | 114 | 13% |
| Ресторан | 23 | 3% |

> **Главный вывод: в 60% случаев агенты сами диагностируют причину на стороне Delever** — это не «агрегатор глючит», а наш интеграционный слой. Это прямой сигнал продукту.

## Корневые причины (из заметок агентов)

| Причина | Инцидентов | Пример |
|---|---|---|
| Вебхук / интеграция не настроена | 106 | «разбираемся с кассовой системой» |
| Несоответствие меню / id позиций | 69 | «количество товара не отправляется в филиал» |
| **Ручное перенаправление заказа** | 41 | агент вручную пробивает заказ |
| Токен / авторизация / ключ | 37 | «продукты недоступны при создании заказа» |
| Филиал / точка / адрес | 29 | заказ уходит не в тот филиал |
| Стоп-лист / неактивная позиция | 18 | «плагин был отключён» |
| Ошибка 500 / задержка | 7 | «ошибка 500 при обработке заказа» |

## Операционная боль

- **Решено в моменте: ~72, открыто/тянется: ~812.** Эти инциденты **долго не закрываются**.
- **41 случай — агенты вручную перенаправляют/пробивают заказ.** Это ручной труд поддержки, маскирующий системный баг: каждый «потерянный» заказ = недовольный ресторан + риск потери реального заказа клиента.

## Рекомендация продукту

1. **Приоритет №1 — надёжность доставки заказа Yandex/Uzum → iiko.** Это 30% всей нагрузки поддержки и 60% вины на нашей стороне.
2. **Топ-3 техпричины к разбору:** (а) вебхук/интеграция не поднимается при онбординге (106), (б) рассинхрон меню/id позиций ломает приём заказа (69), (в) протухшие/неверные токены (37).
3. **Убрать ручное перенаправление** (41) автоматическим ретраем + алертом, пока корневая причина не закрыта.
4. **Метрика для дашборда:** % заказов, не дошедших до POS, по маршруту агрегатор→POS — следить в реальном времени.

_Данные инцидентов: `scripts/pos-incidents.json` (884 записи: агрегатор, POS, симптом, вина, решение)._
